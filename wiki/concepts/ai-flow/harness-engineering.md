## 1. 引言：为什么需要 Harness

### 1.1 AI 辅助开发面临的两个核心问题

过去两年，AI 辅助编码工具——Copilot、Claude Code、Cursor、Devin 等——
迅速从"自动补全"演化到"接管整个任务"。开发者越来越多地把成段代码、
完整功能甚至端到端流水线交给 AI。这种角色变化带来了两个棘手的问题，
它们决定了今天的 AI 辅助开发是"高效"还是"灾难"：

#### 问题一：行为偏离（Behavioral Drift）

AI 在没有明确约束时会做出"看起来合理但项目里不该这么做"的选择：
- 在你的项目里所有 API 调用都走 `request` 封装，AI 却直接 `fetch`
- 项目命名约定是 `kebab-case` 文件名 + `PascalCase` 组件名，AI 全用 `camelCase`
- 已经有了 `<Button>` 组件，AI 重新 `<button styled>` 一遍
- 团队约定错误处理用统一的 `errorBoundary`，AI 却在每个组件里写 try-catch

这些不是 AI 能力问题，而是**缺乏约束机制**——AI 不知道你的"合理"是什么。
每一次违反都在制造技术债：风格不一致、重复实现、约定漂移。
当 AI 参与代码量从 10% 涨到 50%，行为偏离会以指数级累积成不可维护的混乱。

#### 问题二：自主性失控（Autonomy Out of Control）

第二个问题与第一个相反——不是 AI 做得不够好，而是 AI 做得"太多太快"，
人类失去了感知与控制：
- AI 一口气改了 30 个文件，没人知道为什么
- AI 自己运行了 `rm -rf` 类操作，没有审批环节
- AI 部署到了生产，但不知道 spec 是不是被理解对了
- AI 在含糊处自圆其说做出了影响业务的关键决策（"我假设删除是软删除"）
- 测试通过了，但通过的是 AI 自己"理解"的需求，而不是真正的需求

这两个问题指向同一个本质：**AI 需要边界，但不是被关进笼子，而是被装上缰绳。**
让它能跑、跑得快，但跑在正确的方向上，并且在跑出轨道前能被拉回来。

### 1.2 Harness 的定义

> **Harness** 是一套约束体系，让 AI 在**可控的自治状态**下完成软件开发任务。

这个词原本是"马具/缰绳"——给马戴上缰绳不是限制它跑，而是让它跑在正确方向。
转移到 AI 辅助开发的语境，Harness 的目标是：

| 目标 | 含义 |
|------|------|
| **行为可预测** | AI 的产出风格、模式、选择基本一致，不是每次新花样 |
| **质量可验证** | 每一段产出都能被自动化机制检验，不依赖人肉 review |
| **流程可追溯** | 每一个决策、每一次失败、每一次重试都能被回溯 |
| **边界可控制** | 哪些事 AI 能自己做、哪些必须升级到人，事先定义清楚 |
| **失败可恢复** | 出问题时有自动回退，不会把系统带进无法挽回的状态 |

Harness 不是一个工具，也不是一份模板，而是**多个机制协同工作的体系**。
它至少包括四类机制（本文档将在第 7 章详细展开）：

```
┌────────────────────────────────────────────────┐
│  Harness 体系                                    │
│  ┌──────────────────────────────────────────┐   │
│  │ Prompt Harness   — 约束 AI 的"行为"       │   │
│  │ Eval Harness     — 验证 AI 的"产出"       │   │
│  │ Workflow Harness — 约束 AI 的"流程"       │   │
│  │ Autonomy Harness — 约束 AI 的"自主性"     │   │
│  └──────────────────────────────────────────┘   │
└────────────────────────────────────────────────┘
```

四层不是孤立的，而是层层嵌套：自治流程靠工作流定义边界，工作流靠质量检查
保证每步产出，质量检查靠行为约束让 AI 一开始就少出错。四层的完整定义与
依赖关系详见 [Ch7](#7-harness-的四层模型)。

### 1.3 业界发展脉络：从"AI 辅助"到"AI 自治 + Gate"

理解当下的 Harness 思想，需要先看清这两年业界经历的三个阶段：

#### 阶段一：AI 辅助（2022-2023）

代表：GitHub Copilot 早期、Tabnine、CodeWhisperer。

- AI 角色：智能补全、片段生成
- 人的角色：所有决策、所有审查、所有提交
- 约束机制：几乎没有——AI 只看光标周围几十行上下文
- 典型问题：风格不一致、与项目约定脱节、需要大量手动修正

这一阶段没有 Harness 的概念，因为 AI 的"自治度"很低，人随时在场，
风险被人肉兜底。

#### 阶段二：AI 协同（2023-2024）

代表：ChatGPT Code、Claude Code、Cursor Composer、GitHub Copilot Chat。

- AI 角色：完成任务级别的工作（写一个组件、修一个 bug、写一组测试）
- 人的角色：发起任务、确认产出、推进流程
- 约束机制开始出现：CLAUDE.md / AGENTS.md / .cursorrules 等"项目级 prompt"
- 出现新概念：spec-driven development、prompt engineering、context engineering

这一阶段开始重视"上下文工程"——给 AI 足够的项目背景信息，
让它的产出贴合项目实际。但流程仍然是"AI 写 → 人审 → 人决定下一步"，
人是流程驱动者。

#### 阶段三：AI 自治 + Gate（2024-2025）

代表：Devin/Cognition、Copilot coding agent、AWS Kiro、Anthropic agent SDK、各类 IDE 自治模式。

- AI 角色：接管端到端流程——从理解需求到提交 PR、运行 CI、甚至触发部署
- 人的角色：少量 Gate 处的决策与校验，不再事事点头
- 约束机制成熟化：spec 是 single source of truth，confidence threshold、
  permission scopes、auto-rollback、HITL gate 等模式被广泛讨论
- 关键转变：人从"流程驱动者"变成"边界守护者"

这一阶段才真正出现了 Harness 的需求——因为 AI 真的开始"独自跑路"了，
没有约束的自治等于灾难。Harness 的成熟度直接决定了 AI 自治能跑多远。

#### 三个阶段的对比

| 维度 | AI 辅助 | AI 协同 | AI 自治 + Gate |
|------|--------|--------|---------------|
| AI 任务粒度 | 行 / 片段 | 函数 / 组件 / 文件 | 端到端 feature |
| 人介入频率 | 每行 | 每任务 | 每 Gate |
| 约束载体 | IDE 配置 | prompt 文件 | Harness 体系 |
| 主要风险 | 风格不一致 | 上下文丢失 | 自治失控 |
| 关键能力 | 上下文窗口 | prompt engineering | autonomy + verification |

### 1.4 本文档的位置

本文档要解决的问题是**第三阶段**的核心问题：

> **当 AI 真的可以从需求一路跑到部署时，怎么让它跑得既快又稳？**

读完之后，你应该能：
1. 理解 AI 自治流水线的标准三阶段模型（Co-Design → Autonomous Build → Validation）
2. 知道 spec 为什么是分界线、什么样的 spec 才算"定稿"
3. 知道在哪几个 Gate 让人介入、为什么是这几个
4. 能设计自治内核的四件套（confidence threshold / risk tiering / self-healing / DoD）
5. 能识别和避免常见反模式
6. 能评估自己团队当前的自治成熟度，并制定渐进式采纳路径

本文档不绑定任何具体工具，所有原则都用工具无关的语言表达。
真正落地时，你需要把这些原则映射到自己的技术栈——
但只要原则对了，工具是第二顺位的事。

## 2. 核心模型：Spec-Then-Build with Gates

业界经过两年试错，逐步收敛到一个共同结构：**先把需求收敛成一份机器可读、
无歧义的 Spec，然后让 AI 在明确边界内自治执行，只在少数关键节点请人拍板。**
本章给这个结构起名为 **Spec-Then-Build with Gates**，并定义它的三个阶段、
三个 Gate 与一条分界线。后续所有章节都建立在这个模型之上。

### 2.1 三阶段模型

完整流水线分为三个阶段，串联起从模糊需求到上线运行的全过程：

```
┌────────────────────┐    ┌──────────────────────┐    ┌────────────────────┐
│ Phase 1            │    │ Phase 2              │    │ Phase 3            │
│ Co-Design          │ ─► │ Autonomous Build     │ ─► │ Validation         │
│ （人 ⇄ AI 协同）   │    │ （AI 自治执行）       │    │ （终点校验+监控）  │
└─────────┬──────────┘    └──────────┬───────────┘    └─────────┬──────────┘
          │                          │                          │
       Gate 1                     Gate 2                     Gate 3
   Spec 定稿（必经）         高风险触发（罕见）          终点验收（必经）
```

**Phase 1 · Co-Design（协同设计）**
人和 AI 一起把模糊的需求变成精确的 Spec。这个阶段 AI 频繁问问题（ask_user）、
人频繁修订是常态——它的产物就是下一阶段唯一的输入。

**Phase 2 · Autonomous Build（自治执行）**
AI 拿着定稿的 Spec 独立完成实现：拆任务、写代码、跑测试、修缺陷、
合并、部署到预发、灰度推进。整个过程**默认不打断人**，只有信心不足或
触碰风险红线时才升级到 Gate 2。

**Phase 3 · Validation（终点校验与持续监控）**
AI 提交一份结构化的 AutoPilot Report，人对照 Spec 与 DoD 做最终验收。
通过后系统进入持续监控阶段——一旦线上指标异常，触发自动回滚或新一轮迭代。

三阶段的本质区别是**"人介入的频率"**：Phase 1 高频协同 → Phase 2
默认零打断 → Phase 3 仅终点确认。整条流水线的人力投入呈 U 形分布：
两头重、中间几乎为零。

### 2.2 Spec 作为分界线

为什么把分界线放在 Spec 而不是其他位置？答案藏在"AI 适合做什么、不适合做什么"里。

AI 不擅长的事：
- 揣测你的真实意图
- 在多个合理方案之间做价值判断
- 平衡商业、合规、用户体验等多维约束

AI 擅长的事：
- 在明确目标下生成代码
- 跑测试、读日志、调试
- 重复性的工程操作（构建、部署、灰度）

**Spec 恰好是这两类能力的交接点**：
- Spec 之前，问题是"需求是什么、应该长什么样"——价值判断、需要人；
- Spec 之后，问题是"怎么把它造出来、怎么验证、怎么上线"——工程执行、AI 拿手。

把分界线放在 Spec 还有一个工程上的好处：Spec 是一种**可序列化、可版本化、
可 diff、可机器校验**的中间产物。任何"分界线"如果不能被这样处理，
就无法成为流水线两侧的契约。一份模糊的 PRD 不能当分界线，因为它
既不能 diff 也不能机器校验；一段口头共识更不能。能够承担分界线职责的，
只有结构化的 Spec。

> **Spec 不只是文档**：它是一份合同。
> Phase 1 的产出 = Phase 2 的输入 = Phase 3 的验收标准。
> 三阶段共用同一份 Spec，这是整个模型自洽的基石。

### 2.3 三个 HITL Gate

Gate（关卡）是**人必须或可能介入的节点**。三阶段模型只有三个 Gate，
其中两个必经、一个按需触发。这种"少而关键"的设计直接来自业界共识：
HITL gate 太多 → 人成为瓶颈、AI 退化为补全工具；HITL gate 太少 →
失去可控性、出问题时无人兜底。三个是经验上的甜蜜点。

#### Gate 1 · Spec 定稿（必经）

**位置**：Phase 1 末尾、Phase 2 开始之前。
**形式**：人对 Spec 文档显式签字（commit、approve、merge）。
**目的**：锁定下游自治的输入，避免 AI 在错误的 Spec 上跑完整条流水线。
**通过条件**：Spec 满足"机器可判定的定稿标准"（详见 §3.6），
且没有未解决的歧义清单条目。

Gate 1 的关键性怎么强调都不过分——AI 在错误 Spec 上自治得越彻底，
事后修正成本越高。这里多花十分钟，胜过事后回滚一整条流水线。

#### Gate 2 · 高风险触发（罕见）

**位置**：Phase 2 内部，由 AI 主动升级。
**形式**：AI 暂停执行，输出"为什么我打不了这个决定"+ 候选方案 + 推荐项，
请人裁决。
**触发条件**（任一即触发）：
- 信心分数低于阈值（confidence threshold）
- 决策落入高风险层级（删除数据、修改 schema、变更权限、付费动作）
- Spec 在执行中暴露缺陷或矛盾（spec drift）
- 自愈循环连续失败超过预算（self-healing budget exhausted）
- 触碰 permission scope 边界

Gate 2 应该**罕见**——如果它频繁触发，说明 Spec 不够清晰、
risk tiering 设置过紧、或 confidence threshold 设得过高。
频繁的 Gate 2 是模型不健康的信号。

#### Gate 3 · 终点验收（必经）

**位置**：Phase 3 末尾、系统进入"无人值守监控"之前。
**形式**：人对照 AutoPilot Report 做结构化验收（见 §5.1）。
**通过条件**：
- 所有 DoD 条款满足
- 部署后健康检查通过
- 关键业务指标在阈值内
- 人对结果"满意"——这是唯一保留的主观判断

Gate 3 不应是"通读所有代码"，而是"抽查核心结论 + 验证关键指标"。
如果你发现自己每次都在 Gate 3 通读代码，说明你对前两个阶段的
Harness 还不够信任——本质问题在前面，不在 Gate 3。

### 2.4 与传统 SDLC 的对比

为了凸显新模型的不同，下表把它与瀑布、敏捷做横向对比：

| 维度 | 瀑布 | 敏捷 | Spec-Then-Build with Gates |
|------|------|------|----------------------------|
| 主要执行者 | 人 | 人（AI 辅助补全） | AI（人定义 Spec、把守 Gate） |
| 文档地位 | 重，但常与代码脱节 | 轻，依赖口头共识 | Spec 是机器可读契约，是一切 |
| 反馈周期 | 月级 | 周级 | 小时级（自治内部）+ 周级（Gate 间） |
| 人介入频率 | 阶段末评审 | 每日站会、每个 Sprint | 仅 3 个 Gate |
| 失败处理 | 回到上游阶段 | 下个 Sprint 修 | 自愈优先 → 失败则升级或回滚 |
| 可追溯性 | 文档→代码常断 | 依赖 PR 描述 | Feature ID 全链贯通 |
| 适用规模 | 大型、需求稳定 | 中小、需求易变 | 凡是 Spec 可结构化的任务 |

新模型不是要否定瀑布或敏捷的所有理念，而是把它们与 AI 自治能力重新组合：
- 它继承了瀑布的"分界线明确"——Spec 是新的分界线；
- 它继承了敏捷的"快速反馈"——但反馈循环主要发生在 AI 自治内部；
- 它新增了"自治内核"——这是前两者都没有的部分，也是后续章节的重点。

### 2.5 适用范围与不适用范围

Spec-Then-Build with Gates 不是"所有任务都该套上的总模型"。它最适合的是
那些**边界清晰、验证明确、回滚路径存在**的任务；一旦任务本身缺乏这些条件，
强行套进来只会得到一份看似正式、实则失真的 Spec。

更适合 full harness 的任务通常具备四个特征：

- **目标可结构化**：需求能被拆成明确行为、接口契约、DoD
- **结果可验证**：至少有一部分能被测试、探针或指标客观判定
- **风险可分层**：能分清哪些动作低风险、哪些必须升级
- **失败可收敛**：出错后可以回滚、补偿或冻结，不会立刻造成不可逆后果

典型例子包括：明确需求的业务功能、已有先例的重构、契约清晰的 API 改造、
有灰度与回滚能力的常规发布流程。

相反，下列任务不适合直接进入 full harness，至少不适合在无人值守前提下进入：

- **探索性需求**：产品意图尚未收敛，连"做什么"都还在试探
- **强主观设计任务**：大量判断依赖体验直觉、品牌风格、审美取舍
- **一次性排障/抢修**：问题空间不清晰，调试路径高度依赖临场判断
- **不可逆数据动作**：无法安全回滚的迁移、清洗、外部系统写操作
- **开放式研究任务**：目标是发现可能性，而不是按契约交付确定结果

这类任务不是不能用 AI，而是不该直接套用本文档的完整自治模型。更现实的做法是：
**把价值判断部分留在人主导的 Co-Design，把工程执行部分再交给 AI**。如果任务
不能被这样切开，就说明它还没准备好进入 Phase 2。

本章给出了模型的骨架。Ch3 ~ Ch5 分别展开三个阶段的实践，Ch6 讨论横切关注点，
Ch7 把"Harness"这个总概念拆成四层重新审视，Ch8 罗列反模式，
Ch9 给出从 Level 0 到 Level 4 的渐进采纳路径。

## 3. Phase 1：Co-Design（协同设计）

Co-Design 阶段的目标只有一个：**产出一份能让 AI 自治执行的 Spec**。
这个阶段允许人和 AI 频繁来回、允许 AI 主动提问、允许反复修订——
因为它的产出决定了下游所有自治环节的成败。本章详细说明这个阶段的
角色分工、Spec 应该长什么样、如何处理歧义、以及"定稿"该如何判定。

### 3.1 阶段定位

Co-Design 不是"写需求文档"的换皮，它有三个区别于传统需求阶段的特征：

1. **AI 是平等的参与者**：AI 不只是把人写的需求"翻译"成代码，
   而是主动追问"这个边界是什么、那个错误怎么处理、这种输入算什么"。
   人的角色从"自己想清楚再交给 AI"变成"和 AI 一起想清楚"。
2. **产物是 Spec，不是 PRD**：PRD 面向人阅读，留有大量隐含假设；
   Spec 面向 AI 执行，必须显式、结构化、机器可读。
3. **接受多轮迭代**：一个功能从初稿到定稿，5~20 轮对话是常态。
   这里看似"慢"，但每一轮都在为后面的自治节省一次失败回滚。

### 3.2 角色分工

| 角色               | 主要职责                    | 不该做的事                 |
| ---------------- | ----------------------- | --------------------- |
| 人（产品/技术决策者）      | 提供意图、做价值判断、回答歧义、签字定稿    | 替 AI 想实现细节、替 AI 做技术选型 |
| AI（Co-Design 助手） | 追问、提案、整理、把对话沉淀为结构化 Spec | 在歧义未消除时擅自做决定、跳过提问直接动手 |

这个分工的关键是：**人只回答"是什么、为什么"，AI 只追问"怎么定义、
边界在哪"**。当人开始替 AI 想"应该用什么库、怎么拆模块"时，
就回到了"AI 当补全工具"的旧模式；当 AI 在歧义未消除时擅自决策时，
就埋下了 Phase 2 翻车的种子。

### 3.3 Spec 应该包含什么

一份合格的 Spec 至少包含以下七个部分。缺任何一项，下游自治都会
在某个时刻被迫"猜"。

1. **Feature ID 与一句话描述**
   全链路追溯的锚点。所有下游 artifact（PR、commit、测试用例、
   部署记录、监控告警）都通过 Feature ID 串起来。

2. **意图与非目标**
   *意图*：这个功能要解决什么问题、为谁解决。
   *非目标*：明确哪些"看起来该做但本期不做"的事——这是控制范围
   蔓延的关键。AI 在自治时如果遇到"是不是顺手把 X 也做了"的诱惑，
   非目标就是它的刹车。

3. **用户故事 / 行为规约**
   描述"在什么场景下、谁做什么、系统应该如何响应"。
   推荐用 Given-When-Then 形式，因为它天然能转成验收测试。

4. **接口契约**
   所有跨边界的输入输出都要显式：API schema、事件格式、UI 状态机、
   数据模型。AI 在 Phase 2 写代码、写测试、Mock 依赖都靠这个。

5. **约束清单**
   非功能性需求与红线：性能阈值、安全要求、合规约束、依赖限制、
   兼容性要求。这是 Gate 2 的"风险触发器"赖以工作的依据——
   AI 一旦发现某个决策可能违反约束，就立刻升级。

6. **DoD（Definition of Done）**
   一组可机器判定的完成条件：所有验收测试通过、覆盖率达到 X%、
   性能基准在阈值内、文档已更新、监控指标已配置等。
   DoD 是 Phase 2 自终结的判据，也是 Gate 3 验收的清单。

7. **歧义清单（Open Questions）**
   还没敲定的问题列表。Spec 定稿前必须清空，未清空就强行
   通过 Gate 1，等于把决策推迟到 AI 自治期间——而那时 AI
   只能"猜"或"打断人"，两个都是坏选项。

### 3.4 Spec 的形式：人读 + 机读双轨

Spec 必须同时满足"人能读懂"与"机器能解析"两个要求。常见做法：
- **主体用结构化文本**（Markdown + YAML front-matter，或 JSON Schema 引用）
- **接口契约用专门格式**（OpenAPI、JSON Schema、TypeScript 类型、protobuf）
- **行为规约用 Gherkin / 决策表**——可直接生成测试

这种双轨的价值在 Phase 2 才显现：当 AI 想验证"我现在的实现满足
Spec 吗"，它需要把 Spec 喂给某个判定器；当 AI 想生成验收测试，
它需要把 Spec 翻译成测试用例。如果 Spec 只是自然语言段落，
这些自动化都做不到。

### 3.5 歧义处理：ask_user 是 Co-Design 的核心动作

Phase 1 与 Phase 2 在"是否允许 ask_user"这件事上有根本区别：

- **Phase 1 鼓励 ask_user**：每一次提问都是把模糊的意图固化成
  显式约束。问题问得越多越深，下游自治的失败概率就越低。
- **Phase 2 严格限制 ask_user**：只在 Gate 2 触发条件满足时升级。
  日常的"我应该用哪个库、命名是不是这样"绝不打断人。

Phase 1 的 ask_user 应该具备三个特征：

1. **批量提问**：把多个相关问题一次性抛出，而不是一问一停。
   这是对人的注意力的尊重，也是 Co-Design 不退化为聊天的关键。
2. **带候选项与推荐**：不要问"你想怎么处理超时"，而要问
   "超时的处理我倾向 A：返回 503 让客户端重试；B：降级到缓存；
    C：直接 5xx 报错。我推荐 A，因为... 你怎么看？"
3. **追问到根因**：不要满足于第一个答复，要追问"如果 X，那 Y 怎么办"，
   直到边界完全闭合。

### 3.6 Gate 1：Spec 定稿的判定标准

Spec 定稿是 Gate 1 的通过标志。一份 Spec 能称为"定稿"，必须
同时满足以下条件——这个判定**应该尽可能自动化**，由 lint/校验脚本完成：

- ✅ 七大部分齐备（§3.3 的 1~7 项）
- ✅ 所有接口契约通过 schema 校验
- ✅ DoD 条款全部可机器判定（不允许"用户体验良好"这种 DoD）
- ✅ 歧义清单为空
- ✅ Spec 已 commit 到版本管理，有明确版本号
- ✅ 有人显式签字（PR approve / 文档 sign-off）

> **签字 ≠ 走过场**。签字代表你愿意在 Phase 3 用这份 Spec 做验收。
> 如果你签字时心里仍有"AI 应该懂我意思"的依赖，等于把判断责任
> 转嫁给一个根本不了解你完整意图的执行者——这是 Spec-Then-Build
> 模型最常见的失败模式。

### 3.7 反复迭代不是浪费

新手最常见的疑虑是："Phase 1 这么慢，AI 加速的意义在哪？"

要算总账：
- 旧模式：人写 PRD 1 天 → AI 写代码 0.5 天 → 来回返工 5 天 → 上线
- 新模式：Co-Design 2 天（Spec 定稿） → 自治执行 0.5 天 →
  Validation 0.5 天 → 上线

新模式的"慢"集中在 Phase 1，但 Phase 2 几乎不需要人在场——
人的总时间反而下降，更重要的是：**人的注意力得以连续地用在
"想清楚要什么"上，而不是被打断成碎片去 review 代码细节**。

Phase 1 投入的每一分钟，都是用来换 Phase 2 的"无打断时长"。
这个折算关系要内化为团队共识，否则 Co-Design 阶段会被反复压缩，
最终模型退化为"PRD + AI 补全"。

## 4. Phase 2：Autonomous Build（自治执行）

Autonomous Build 是整个模型的核心章节，也是最容易翻车的阶段。
这里的"自治"不是"放任"——它由一整套机制护着 AI 在边界内独立工作。
本章把这些机制拆成八个支柱：每个支柱单独看都不复杂，但缺任何一个，
"自治"都会退化为"放生"。

### 4.1 阶段定位与默认行为

进入 Phase 2 时，AI 拿到的输入只有两样：**已定稿的 Spec** 和
**当前代码库快照**。从这一刻起，到产出 AutoPilot Report 为止，
AI 的默认行为是：

- 不向人提问（除非命中 Gate 2 触发条件）
- 不超出 Spec 范围"顺手"做额外的事（非目标守则）
- 所有决策、动作、失败、自愈都被结构化记录
- 失败优先尝试自愈，自愈失败再升级

这个"默认不打断人"是 Phase 2 区别于补全工具的本质特征。
团队在采纳期最容易破坏它的方式是"人忍不住主动看一眼、改一下"——
一旦人开始介入，AI 就会调整自己以迎合人的偏好，模型的"自治"
属性立刻失效。**Phase 2 期间，人最该做的事是离开屏幕。**

### 4.2 八大支柱概览

```
┌─────────────────────────────────────────────────────────────┐
│            Phase 2 · Autonomous Build 八大支柱              │
├─────────────────────────────────────────────────────────────┤
│ ① Plan-First Execution      明确步骤、可被审查的执行计划      │
│ ② Continuous Verification   每一步都有自验证                  │
│ ③ Confidence Threshold      信心不足主动停                    │
│ ④ Risk Tiering              不同风险走不同流程                │
│ ⑤ Self-Healing Loop         失败优先自愈，有预算上限          │
│ ⑥ Permission Scopes         能力按需授予，最小必要            │
│ ⑦ Gradual Env Promotion     按环境逐级推进                    │
│ ⑧ Observable Trace          全程结构化可观测                  │
└─────────────────────────────────────────────────────────────┘
```

下面逐一展开。

### 4.3 支柱 ① · Plan-First Execution（计划先行）

AI 进入 Phase 2 的第一个动作不是写代码，而是**输出一份执行计划**：
把 Spec 拆成有序、有依赖、可单独验证的子任务列表。

一份合格的执行计划包括：
- **任务节点**：每个节点有明确的输入、输出、验证手段
- **依赖关系**：哪些任务必须串行、哪些可并行
- **风险标注**：每个节点对应哪个风险层级（见 §4.6）
- **预估自愈预算**：复杂节点允许多少次失败重试

执行计划不是给人 review 的（如果需要 review，本质属于 Phase 1
没收敛好），它是 AI 自己的"路线图"——也是事后追溯"AI 为什么
这样做"的入口。

> **为什么计划要先于代码？** 没有计划的执行是"流式生成"——
> AI 一边写一边想，错误会在长链路里悄悄累积，很难被发现。
> 有计划的执行是"图式生成"——每一步都有明确的"我现在在哪、
> 下一步去哪、这一步成功的标志是什么"，错误能被立刻定位。

### 4.4 支柱 ② · Continuous Verification（持续验证）

AI 的每一步都必须有"我做对了吗"的客观判定，而不是"我觉得我做对了"
的主观陈述。常见的验证手段（按层级递增）：

| 层级 | 验证手段 | 例子 |
|------|---------|------|
| L0 静态 | 类型检查、Lint、格式化 | `tsc --noEmit`、`ruff` |
| L1 单元 | 单元测试 | `jest`、`pytest` |
| L2 契约 | 接口契约测试 | OpenAPI 校验、Pact |
| L3 集成 | 集成 / 端到端测试 | Playwright、Cypress、k6 |
| L4 行为 | 真实环境健康检查 | 探针、SLO 探测 |
| L5 业务 | 关键业务指标 | 转化率、留存、客单价 / GMV |

Phase 2 的核心规则是：**任何代码变更，至少跑通到对应层级的验证才能继续。**
比如改了 API schema，最低跑到 L2；上了预发，最低跑到 L4；进灰度，
必须跑 L5。AI 不能"自己说过了"——必须有可被引用的验证产物
（测试报告、探针响应、指标快照）。

这条规则的反面就是 Anti-pattern：让 AI"声称完成"而不强制验证
（详见 §8）。

### 4.5 支柱 ③ · Confidence Threshold（信心阈值）

AI 在每一个决策点都需要给出一个"信心分数"。信心高 → 直接执行；
信心低 → 触发 Gate 2 升级。

信心分数不是玄学，可以由以下信号合成：
- **Spec 覆盖度**：当前决策是否完全被 Spec 覆盖？
- **历史相似度**：仓库里有相似的先例吗？
- **测试可达性**：这个改动能被现有测试覆盖吗？
- **副作用面积**：影响多少模块、多少行代码？
- **回滚成本**：万一错了，撤销有多难？

阈值需要按团队风险偏好校准，但有一个铁律：**宁可信心阈值偏高
导致 Gate 2 多触发一次，也不要让低信心决策悄悄通过。**

阈值过高的代价是"人累一点"；阈值过低的代价是"线上出事"。
两者不对称。

#### 信心阈值的工程化落地

这里最容易犯的错，是把"信心分数"理解成模型随口报出的一个数字。
**真正可用的 confidence 不是自我感觉，而是基于证据的升级规则。**

落地时建议遵循三个原则：

1. **先用离散档位，再追求精细分数**：很多团队一开始并不具备
   校准 0.73 和 0.81 区别的能力。先用高 / 中 / 低三档，往往比
   假装精确更可靠。
2. **每次低信心都要能解释原因**：例如"Spec 没覆盖删除语义"、
   "仓库中没有相似先例"、"现有测试无法触达这个改动"。说不出原因的
   低信心，和说不出原因的高信心一样不可用。
3. **阈值必须用历史事件回校**：Gate 2 的升级记录、回滚事件、
   自愈失败事件，都是调整阈值的样本。如果低信心升级后总被证明是多余，
   阈值可以略放；如果线上问题大多来自"当时其实不该放行"，
   阈值就该收紧。

在系统还不成熟时，可以用一个保守的经验法则兜底：**同时满足
"缺少 Spec 覆盖 + 缺少仓库先例 + 缺少可达验证" 三项中的任意两项，
默认视为低信心**。不要等模型"觉得自己不确定"才升级——
那通常已经太晚。

#### 一个最小可用的信号矩阵

与其让模型报一个空口的数字，不如让它在每个决策点填下面这张表，
然后用固定规则合成档位。这是目前落地门槛最低的方案：

| 信号 | 取值 | 说明 |
|------|------|------|
| Spec 覆盖 | 是 / 部分 / 否 | 当前决策是否被 Spec 某条显式覆盖（要能给出引用位置） |
| 仓库先例 | 是 / 否 | 是否存在"同模块 / 同模式"的历史改动可参照 |
| 测试可达 | 是 / 否 | 现有自动化验证是否能触达这次改动的行为 |
| 影响面 | 小 / 中 / 大 | 改动波及的模块数 / 文件数是否在预期内 |
| 回滚成本 | 低 / 中 / 高 | 万一错了，撤销是 revert commit、还是需要补数据 / 外部动作 |

合成规则示例（团队可按风险偏好调）：

- **高信心**：Spec 覆盖=是 且 测试可达=是 且 回滚成本≤中
- **低信心**：Spec 覆盖=否 或 回滚成本=高 或（仓库先例=否 且 测试可达=否）
- **其余**：中信心（T2 及以下继续执行；T3 强制走加重验证；T4 一律升级）

这张表的价值不在"精确"，而在**每次低信心都能被解释**——每个字段的取值
都是事实性判断，不是主观评分。阈值可以调，但判断依据必须留痕到 Trace 中。

### 4.6 支柱 ④ · Risk Tiering（风险分层）

不是所有动作都该走同样的流程。把动作按"出问题的代价"分层，
不同层走不同的护栏：

| 风险层 | 例子 | 护栏 |
|-------|------|------|
| T0 只读 | 跑测试、读日志、查文档 | 无限制 |
| T1 局部写 | 改单个模块代码、加测试 | 自动执行 |
| T2 跨模块 | 重构接口、改公共类型 | 自动执行 + 强制 L2/L3 验证 |
| T3 数据/配置 | 改 schema、配置变更 | 强制双签（AI + AI 二次自检） + L4 验证 |
| T4 不可逆 | 删数据、删表、生产 hotfix、付费动作 | 必须升级 Gate 2 |

T0~T2 是 Phase 2 的常态，AI 全程自治。T3 加重验证但仍自治。
**T4 一律升级**——这是 permission scope 的硬约束，不接受
"我有信心"的豁免。

风险分层的好处是：把"是否升级"从"AI 主观信心"变成"客观风险层 +
信心阈值"的双重判定，避免 AI 在高风险动作上自我说服。

风险分层也不该按动词名字面理解，而应该按**后果**判断。比如"改配置"
在 dev 里可能只是 T1，在生产环境里却可能是 T3；"发起外部 API 请求"
如果只读查询可能是 T1，但如果会真正扣费或写第三方系统，就应直接上升。

一个实用的判定法是只看三件事：

- **影响面**：会波及多少用户、多少模块、多少环境
- **可逆性**：做错后能否可靠回退，还是只能补救
- **外部副作用**：是否会影响数据库、权限、计费、第三方系统

这三个维度里，只要有一个明显偏高，就不应该继续按低风险处理。

### 4.7 支柱 ⑤ · Self-Healing Loop（自愈循环）

Phase 2 必然遇到失败：测试挂了、构建报错、部署回滚、监控告警。
AI 应该**优先尝试自愈**，而不是立刻升级人——但自愈必须有预算上限，
防止"无限重试黑洞"。

#### 自愈与 Gate 2 的先后顺序

自愈和 Gate 2 都是"失败/不确定"的响应，二者必须有明确的次序，否则
容易出现"一边自愈一边又升级"的混乱：

1. **失败发生时先看初始信心**：如果修复假设本身就低信心（见 §4.5 信号矩阵），
   **跳过自愈、直接升级 Gate 2**——在低信心上重试只会放大事故面
2. **修复假设足够高信心时才进入自愈循环**：按下面的标准结构跑
3. **任一预算耗尽**（单点 / 总量 / 时间）→ 升级 Gate 2，不允许"再试一次"
4. **自愈过程中风险层升级**（例如修复路径意外需要改 schema）→ 立刻中断
   自愈、升级 Gate 2，不以"我已经在自愈中了"为由继续

换句话说：**自愈是"高信心 + 低风险 + 有预算"三个条件的交集**，
任一条件不满足都应退出自愈、交由 Gate 2 决策。

#### 自愈循环的标准结构

```
失败 → 收集证据（日志、堆栈、diff、监控曲线）
      ↓
      诊断（这是什么类型的问题？）
      ↓
      生成修复假设 + 信心分数
      ↓
   信心 ≥ 阈值？
      ├─ 是 → 执行修复 → 重跑验证 → 成功？退出 / 失败？回到顶部
      └─ 否 → 升级 Gate 2
```

预算上限的常见设置：
- **单点预算**：同一个失败点，最多自愈 N 次（典型 N=3）
- **总预算**：整个 Phase 2，自愈总次数不超过 M（典型 M=10）
- **时间预算**：自愈耗时不超过原计划的 K 倍（典型 K=2）

任何一个预算耗尽，立刻升级 Gate 2，不允许"再试一次"。
这条规则是防止 AI 在错误方向上越跑越远的最后防线。

#### 自愈的适用边界

自愈最适合的是**确定性、局部性、可重复验证**的问题：例如编译失败、
测试断言错误、依赖版本不匹配、配置值遗漏、探针未通过等。

相反，下面这些场景不应该被当作普通自愈任务：

- 需要改变业务语义或补做产品决策
- 涉及 schema、数据清洗、权限策略、计费逻辑
- 已经影响外部系统，且没有明确补偿路径
- 失败原因本身不明确，只能靠试错不断扩大修改面

一旦进入这些区域，继续"再修一次看看"通常不是自愈，而是在放大事故面。
**自愈的前提不是 AI 足够勤奋，而是问题本身足够收敛。**

### 4.8 支柱 ⑥ · Permission Scopes（权限作用域）

AI 在 Phase 2 拿到的能力必须是**最小必要集**，而不是"管理员权限随便用"。
权限作用域至少要在三个维度上做隔离：

1. **代码维度**：能改哪些目录、不能改哪些目录（如 `infra/`、
   `migrations/` 默认禁写，要改必须 Gate 2 升级）
2. **环境维度**：能部署到哪些环境（dev 默认放开，staging 受限，
   prod 禁止直接部署）
3. **外部资源维度**：能调哪些 API、能花多少钱、能发多少邮件
   （所有付费动作走 T4，必须升级）

权限作用域是 risk tiering 的执行机制——风险层只是分类，
作用域才是真正的"红线"。一个动作即使被 AI 认为是 T2，
只要它触碰了禁写目录，就必须升级。

### 4.9 支柱 ⑦ · Gradual Env Promotion（环境逐级推进）

AI 不应该一次把代码送到生产。标准的环境梯度：

```
local CI → dev 环境 → staging 环境 → 灰度（金丝雀） → 全量生产
   ↓          ↓             ↓              ↓                ↓
  L0~L1      L0~L3        L0~L4         L0~L5            持续监控
```

每一级有明确的**入门条件**与**晋级条件**：
- 入门条件：上一级所有验证通过
- 晋级条件：本级运行至少 X 时间，所有指标在阈值内

灰度阶段尤其关键。AI 应该：
- 控制灰度比例（从 1% 起步逐步放大）
- 监控关键业务指标（不仅是技术指标）
- 一旦超阈值，**自动回滚**而不是升级 Gate 2——回滚是 T1 操作，
  比升级更快、更安全

> **对于允许 AI 自治推进灰度的变更，自动回滚是强制能力——前提是
> 回滚本身安全、可验证、不会制造更大副作用**。如果某类变更做不到
> 这一点（例如伴随不可逆 schema 迁移、已发出的外部写请求、无法幂等
> 重放的异步任务），那问题不在"要不要自动回滚"，而在于**这类变更
> 不该进入无人值守灰度**，应直接标记 T4、走 Gate 2。

### 4.10 支柱 ⑧ · Observable Trace（可观测轨迹）

Phase 2 的所有动作都必须留下结构化轨迹：决策、命令、产物、
验证结果、失败、自愈、升级。这条要求至少服务三件事：

1. **事后审计**：出问题时能精确定位"哪一步、为什么、AI 当时
   看到了什么"
2. **训练反馈**：成功与失败的轨迹是改进 prompt、调整阈值、
   优化 Spec 模板的素材
3. **Gate 3 验收依据**：AutoPilot Report 不是 AI 的"自夸"，
   而是从轨迹自动汇总出来的事实陈述

轨迹应该满足：
- **结构化**：JSON / 事件流，可被脚本查询，不是纯文本日志
- **不可篡改**：写入即冻结，AI 不能"美化"自己的历史
- **链路完整**：每条记录能反向追溯到 Feature ID 与 Spec 条款
- **包含足够上下文**：保留 prompt 版本、context 引用、工具调用元数据；
  原始敏感内容默认不直接入轨迹

无可观测轨迹的"自治"等于"暗箱"。一旦出事，团队会本能地缩回到
"以后还是别让 AI 自己做了"——而问题的根源不在 AI，
而在没法看清 AI 做了什么。

#### Trace 不是"全量落盘"

"可观测"不等于"把一切都永久保存"。如果团队把所有 prompt、
上下文、日志、请求体、工具输出原封不动地写入轨迹，新的风险会立刻出现：
密钥泄漏、PII 暴露、合规越界、审计面失控。

更稳妥的做法是分层留痕：

- **默认层**：记录决策类型、触发规则、命令摘要、产物引用、验证结果
- **审计层**：只在需要时保留脱敏片段，用于事故追查
- **原始层**：极少数高敏日志短期留存，受单独权限控制，并设置自动过期

治理要求至少包括：

- **敏感信息默认禁入**：token、cookie、密钥、身份证号、原始支付数据
  不应直接写入 Trace
- **访问分级**：不是所有开发者都能看完整轨迹，尤其是审计层和原始层
- **保留策略**：不同层级有不同保存时长，避免"永远留着以后再说"
- **引用优先于复制**：能记录 artifact ID、版本号、对象存储地址的，
  不要把整段原文重复拷进轨迹

Trace 的目标是让事故可还原，而不是让数据治理失控。

### 4.11 八大支柱的相互关系

八个支柱不是清单，而是相互咬合的护栏：

- ① Plan + ⑧ Trace → 让"AI 在做什么"可见
- ② Verification + ⑤ Self-Healing → 让"AI 做错了能自己修"
- ③ Confidence + ④ Risk Tiering → 让"AI 知道什么时候该停"
- ⑥ Permission + ⑦ Env Promotion → 让"AI 即使想越界也越不出去"

任何一根支柱缺失，都会让其余七根的价值打折。比如：
- 有 Plan 但没 Trace，事后审计失败
- 有 Verification 但没 Self-Healing，每个失败都打断人
- 有 Confidence 但没 Risk Tiering，高风险动作靠主观信心通过
- 有 Permission 但没 Env Promotion，权限再小也可能一次性影响生产

设计 Phase 2 时，应该把八根支柱当成 **Level 3+ 完整自治的最低集**，
而不是"先做几个看看效果就宣称已经安全"。在采纳路径上，你当然可以
逐步建设其中的若干支柱；但在没补齐之前，**只能把自治范围限制在
对应成熟度内**，不能把半套护栏包装成 full autonomy。

换句话说：**渐进采纳是允许的，误判成熟度是不允许的**。残缺的护栏
之所以危险，不是因为它们不能存在，而是因为团队太容易在它们之上产生
"已经可以无人值守"的错觉。

### 4.12 Phase 2 终结：AutoPilot Report

Phase 2 终止于 AI 输出 AutoPilot Report，作为 Phase 3 的输入。
Report 不是 AI 写的散文，而是从 Trace 中**结构化汇总**出的事实清单：

- Feature ID 与 Spec 版本
- 执行计划与每个节点的实际状态
- 所有验证结果（按层级）
- 所有自愈事件（次数、原因、修复方式）
- 所有 Gate 2 升级事件（如有）
- 最终部署位置与监控配置
- DoD 逐条核对结果
- 已知遗留问题与建议

Report 的作用是把 Gate 3 从"通读代码"变成"核对事实"。
人看的不再是"AI 做得对不对"，而是"AI 自报的事实是否一致、
DoD 是否真的满足、有没有自己回避的坑"。

#### AutoPilot Report 的最小 Schema 示例

为了保证 Report 是"从 Trace 渲染"而不是"AI 自由叙述"，建议
把字段固化成一份 schema。下面是一个可直接照抄的最小骨架（YAML 形式，
JSON / protobuf 同构）：

```yaml
feature_id: FEAT-2025-0137
spec_version: "1.4.0"           # 与 Gate 1 签字版本对齐
spec_ref: "specs/FEAT-2025-0137.md@a1b2c3d"

plan:
  nodes:
    - id: N1
      title: "新增 /orders 查询接口"
      risk_tier: T2
      status: done               # done / failed / skipped
      artifacts: [pr/482, commit/9f3a1c]

verifications:                   # 对应 §4.4 的 L0~L5
  L0: { status: pass, ref: ci/run/88421 }
  L1: { status: pass, ref: ci/run/88422, coverage: 0.87 }
  L2: { status: pass, ref: ci/run/88423 }
  L3: { status: pass, ref: e2e/run/551 }
  L4: { status: pass, ref: probe/staging/2026-04-23T10:00Z }
  L5: { status: pending, window: "24h", ref: dashboard/feat-0137 }

self_healing:
  events:
    - at: N1
      attempts: 2
      cause: "lint fail on import order"
      resolution: "auto-fixed via eslint --fix"
  budget_used: { per_node: 2, total: 2, time_ratio: 1.1 }

escalations:                     # Gate 2 升级记录；无则空数组
  - at: N3
    reason: "low_confidence"
    signals: { spec_coverage: partial, rollback_cost: high }
    resolved_by: "human:alice@2026-04-23T11:12Z"
    decision: "approved with scope narrowed to read-only"

deployment:
  envs: [dev, staging, canary-1%]
  rollback_armed: true
  monitors: [error_rate, p95_latency, checkout_conversion]

dod:                             # 逐条对应 Spec.DoD
  - id: DOD-1
    text: "所有新接口返回 200 或 4xx，无 5xx"
    status: pass
    evidence: probe/staging/2026-04-23T10:00Z
  - id: DOD-2
    text: "覆盖率 ≥ 80%"
    status: pass
    evidence: ci/run/88422#coverage

open_issues: []                  # AI 自报的已知遗留；不允许隐藏
```

关键约束：

- **所有字段都是引用，不是叙述**：`ref` 必须指向可回溯的 artifact
- **AI 不允许增删字段**：schema 由 Harness 维护，不在 PR 里自行扩展
- **`status` 是枚举值**：不接受 "mostly passed" 这类自由文本
- **`open_issues` 为空不等于没问题**：Gate 3 会对照 Trace 核对有没有被掩盖

这份 schema 的意义不是"好看"，而是让 Gate 3 的核对清单（§5.1）可以
完全脚本化——每个核对项都能直接 query 一个字段。

下一章展开 Phase 3 如何使用这份 Report。

## 5. Phase 3：Validation（终点校验与持续监控）

Phase 3 是流水线的"出口"。这里只做两件事：**对照 Spec 验收
AutoPilot Report**，以及**进入持续监控、为下一轮迭代闭环**。
这一阶段的核心理念是"**人确认事实，不复核执行细节**"。

### 5.1 Gate 3 验收：核对事实，不通读代码

很多团队在采纳初期会本能地在 Gate 3 通读代码、跑一遍 demo、
让 QA 全量回归——这本质上是对 Phase 2 不信任。这种"补做"
有两个坏处：

1. **它把 Gate 3 变成了瓶颈**，自治带来的速度优势被吃光
2. **它给了团队"反正最后会有人 review"的依赖心理**，反过来
   削弱了对 Phase 1/2 Harness 的投入

更好的姿势是把 Gate 3 拆成**结构化的核对清单**，针对
AutoPilot Report 做事实校验：

| 核对项 | 来源 | 通过条件 |
|--------|------|---------|
| Feature ID 链路 | Report → PR / commit / 部署记录 | 全链贯通、无孤儿[^1] |
| Spec 版本一致 | Report 引用的 Spec 版本 | 与 Gate 1 签字版本一致 |
| DoD 逐条核对 | Report 的 DoD section | 每条有验证产物链接 |
| 验证层级覆盖 | Report 的 verification section | T1~T4 风险层动作有对应层级证据 |
| 自愈与升级事件 | Report 的 healing / escalation section | 数量与原因合理、无被掩盖的失败 |
| 灰度指标 | Report 的 monitoring section | 关键业务指标在阈值内 |

这份清单上的每一项，**人只需要"看是否齐全 + 数据是否一致"**，
不需要重新执行。如果某一项不齐全或不一致，问题不是"AI 做错了"
而是"AI 隐瞒了"——这是更严重的信号，应该回 Phase 2 或反思 Spec。

[^1]: **孤儿**指链路中找不到反向对应的 artifact：例如一个部署记录
找不到对应 PR，或一个 commit 无法追溯到 Spec 条款。孤儿出现意味着
Feature ID 链路在某个环节被绕过，事后追溯会在此断裂。

### 5.2 主观判断：留给人的最后一寸

Gate 3 必须保留**一个**主观判断点：**这个产出是否符合我对功能的整体直觉？**

举例：Spec 说"用户提交后跳转列表页"，AI 实现了跳转，DoD 也满足，
但你打开界面后觉得"跳转后用户会困惑、应该有一个 toast 提示"——
这种感受没有出现在 Spec 里，但你的直觉是对的。

这一寸主观判断有两个用途：
- 当下：决定是否需要补一轮迭代
- 长期：作为下一次 Co-Design 的"经验输入"——你刚才的直觉
  应该被沉淀进 Spec 模板，让下次类似功能在 Phase 1 就被覆盖

注意"一寸"——它必须很小。如果你在 Gate 3 的主观判断展开成
"我还想加 A、改 B、调 C"，那不是 Gate 3 的事，而是新一轮
Phase 1 的需求。**当场改不进当前 Spec**——硬塞进去会破坏 Spec 是
"Phase 1 ⇄ Phase 2 ⇄ Phase 3 共用契约"的基石。

### 5.3 持续监控：Gate 3 之后

Gate 3 通过 ≠ 任务结束。任务真正"完成"的时刻是**线上指标稳定
运行了一个监控窗口**。这个窗口期内，AI 仍持续值守：

- 监控关键技术指标（错误率、延迟、资源消耗）
- 监控关键业务指标（转化、留存、客单价等领域相关指标）
- 监控 Spec 中显式列出的"成功标志"
- 任一指标越线，**自动触发回滚 + 创建新一轮 Co-Design 任务**

监控期通常按风险层级设定：T1/T2 类变更可能 1~24 小时，T3/T4
甚至需要数天。在这个期间，AI 不被认为"已完成"，团队不应该
立刻开新功能压上去——否则一旦回滚，多个变更纠缠在一起，
责任难以剥离。

### 5.4 闭环：把 Phase 3 的发现喂回 Phase 1

Spec-Then-Build 不是单向流水线，它要形成闭环。Phase 3 产出的
经验应该结构化地回流到 Phase 1：

- **DoD 不充分**：Gate 3 发现 DoD 漏了某项 → 更新 Spec 模板的 DoD checklist
- **Gate 2 误升级**：Phase 2 升级的问题其实 Spec 该说清楚 → 增补
  Spec 七大部分的对应栏目
- **监控漏报**：线上出问题但监控没抓到 → 把对应指标列入 Spec 默认监控集
- **主观判断反复出现**：Gate 3 反复在某类问题上动用直觉 → 把它
  转成显式 DoD 条款

闭环的本质是：**Harness 自己也是被持续改进的对象**。一份 Spec 模板、
一组 DoD 标准、一套风险分层、一组监控指标都不是天降的最佳实践——
它们是从过往任务的 Gate 3 反思中长出来的。**没有闭环的 Harness
会停滞，停滞的 Harness 一定会被绕过。**

### 5.5 Phase 3 的反模式速览（详见 Ch8）

- ❌ Gate 3 重新通读代码 / 跑全量回归
- ❌ Gate 3 现场加需求、临时改 Spec
- ❌ Gate 3 通过即下线监控
- ❌ Phase 3 的发现没有回写任何模板
- ❌ AutoPilot Report 由 AI 自由叙述、不是从 Trace 汇总

## 6. 横切关注点

有一些主题不属于任何单一阶段，而是贯穿整条流水线。把它们放在一章
集中说明，可以避免在每章重复，也方便团队在搭建 Harness 时
逐项核查"我有没有漏配"。

### 6.1 Feature ID：贯穿全链的追溯锚

Feature ID 在 Phase 1 创建，在 Phase 2 被所有 artifact 引用，
在 Phase 3 被验收清单核对。它的作用是让"一个功能"在整条
流水线上有**唯一可追溯的身份**：

- 一份 Spec → 一组 PR → 一组 commit → 一组测试用例 → 一次部署
  → 一组监控指标，全部通过 Feature ID 索引

实现层面通常做法：
- Spec 文件名 / front-matter 包含 ID
- commit message / PR 标题强制带 ID（CI 校验）
- 测试用例命名包含 ID
- 部署 release tag、监控仪表盘 label 都带 ID

Feature ID 不是"管理流程的形式主义"——它是事后追溯
"为什么 AI 这样改、源头需求是什么"的唯一可靠路径。
没有它，Trace 再详细也是孤岛。

### 6.2 Spec 与代码的双向 drift 检测

Spec-Then-Build 模型最大的运行时风险是**"Spec 与代码失去同步"**：
- Spec 改了但代码没跟上 → AI 在错误的契约上自治
- 代码改了但 Spec 没跟上 → 下次自治时 AI 看到的"事实"是旧的

防御手段：
- **CI 强制校验**：每个 PR 必须显式声明"是否影响 Spec"，影响则必须
  附带 Spec 的更新 diff
- **机器可比对**：接口契约部分用 schema 描述，CI 自动对比代码与 Spec
- **定期反向扫描**：周期性任务由 AI 扫描代码，输出"代码与 Spec
  的不一致清单"，作为 Phase 1 闭环输入

drift 是熵增问题，必须有"反向回流"机制持续对冲，否则三个月后
你的 Spec 库就会变成"曾经的需求快照集合"而不是"当前事实"。

### 6.3 Context 工程：让 AI 知道它该知道的

AI 在每个阶段拿到的 context 不是它一次性"消化"的全部信息，
而是它在每个决策点能看到的信息。Context 工程的目标是让 AI 在
**该知道某事时恰好知道、不该被分心时不被淹没**。

实践要点：
- **分层 context 文件**：仓库根目录的"AGENTS 入门"、模块级的
  "本模块约定"、临时任务的"本次约束"
- **Spec 直接引用相关代码段**：避免 AI 自己"猜在哪"
- **历史决策可检索**：过去的 Decision Log 应该是 AI 可读的，
  不是埋在 wiki 里
- **避免"塞满 context window"**：相关性比信息量更重要，
  长尾 noise 会让 AI 关注力发散

Context 工程不是 Phase 1 或 Phase 2 的事，它是横切的——
每个阶段都在持续地"用 context 引导 AI 做对的决策"。

### 6.4 Prompt 与策略的版本化

AI 的行为由 prompt + 策略（阈值、风险层、权限作用域）共同决定。
这些都应该和代码一样**版本化、可 review、可回滚**：

- Prompt 文件进 git，改动走 PR
- 风险阈值、信心阈值、自愈预算等数值进配置文件，不是硬编码
- 重大策略调整应有 changelog，说明"为什么改、影响哪些任务"

没有版本化的 prompt = 没有版本化的源代码。一旦发生事故，
你既不知道"AI 当时看到的指令是什么"，也不知道"事故后我改了什么、
有没有引入新问题"。

### 6.5 失败的"广播"与组织学习

AI 自治有一个微妙的副作用：失败容易"无声"——AI 自愈过、
自动回滚了、Gate 3 通过了，团队其他成员可能完全不知道
"这个功能曾经在 Phase 2 触发过 5 次自愈"。

健康的做法：
- AutoPilot Report 周期性汇总成"团队周报"自动广播
- 高频自愈点、反复升级的 Gate 2 触发器，作为团队复盘议题
- "Spec 模板更新"应在团队层面公告，而不是某个人静悄悄改

失败若不广播，组织学不到东西；学不到东西，下一次同样的失败
还会发生。**Harness 是工具，组织学习才是它的复利**。

### 6.6 安全与合规的特殊待遇

涉及安全、合规、隐私、计费的动作必须**永远在 T4**，无论 AI
信心多高、Spec 写得多详细。这条不接受任何"局部豁免"。

理由很简单：这类动作出错的代价不是"修一下"，而是品牌、法律、
财务损失——这些损失没法靠 Phase 2 的 Verification 在事中捕捉，
只能靠"事前必须有人拍板"挡住。

落地形式：
- Permission Scope 把这些动作所在的目录 / API 列入 hard-deny
- 即使 AI 在 Phase 2 的子任务里需要它们，必须升级 Gate 2
- Gate 2 通过时，授权应该是**单次、最小范围、即过期**的，
  而不是"以后这一类都允许"

#### T4 内部的细分必须走治理流程，而不是 inline 豁免

实战中会遇到"T4 太粗"的诉求：例如"低额度、幂等、可对账的扣费"
和"任意金额、即时到账的扣费"显然不是同一等级的风险。这类细分**可以做**，
但有一个硬性约束：

> **T4 内部的子分层必须走专门治理流程**（安全/合规/财务多方签字、
> 留档成文），**不接受在 PR 或任务中"inline 豁免"**——
> 即不允许某次执行时以"这次金额小 / 这次只改 dev schema"为由
> 临时把动作降级成 T3。

理由是 T4 的本质风险在"错一次就造成不可逆损失"，而 inline 豁免
恰好是"让错一次的概率变大"的机制。把细分沉淀成受治理的策略文档
（可被审计、可被回滚整条策略），是在保留灵活性的同时不放弃兜底。

### 6.7 多 AI 协作时的边界

当你有多个 AI Agent（比如规划 Agent + 实现 Agent + 测试 Agent）时，
本文档的所有原则仍然适用，但要额外注意：

- **每个 Agent 都受同一套 Spec 约束**，不能各自解读
- **Trace 必须能区分"哪个 Agent 做了什么"**，不能合并成一锅粥
- **Agent 之间的输入输出也是 Spec**——Agent A 给 Agent B 的不能是
  自然语言段落，必须是结构化契约
- **升级 Gate 2 时由谁去升级**要预先约定，避免"互相推诿"

#### 多 Agent 特有的三类风险

单 Agent 下不存在、但多 Agent 下会显著放大的问题主要有三类，
每一类都需要专门的机制兜底，而不是寄希望于"大家都遵守 Spec"：

**① 并发冲突（Concurrent Write Conflict）**
多个 Agent 可能同时修改同一份代码 / 配置 / 文档。仅靠 git merge
不够——语义冲突（同一函数两个方向的重构）不会触发 git 冲突，
但结果一样是坏的。

- 对策：在任务分发层引入**资源锁 / 分片**（按目录、按模块、按文件前缀），
  同一资源同一时间只能由一个 Agent 持有写权限
- 对策：合并前必须过一道**跨 Agent diff review**，由独立的仲裁 Agent
  或人工核对是否存在语义冲突

**② 失败归因（Blame Attribution）**
出问题时，"是 Planner 拆错了任务、还是 Implementer 实现偏了、
还是 Tester 漏验了？"如果 Trace 没做好，三个 Agent 互相指证会
消耗大量调查时间。

- 对策：Trace 的每条记录必须带 `agent_id` + `parent_decision_id`，
  能反向还原整条决策链
- 对策：每个 Agent 的输出契约里显式包含"我依赖的上游结论是 X"，
  这样事后可以追查到底是谁先犯错

**③ 级联自愈（Cascading Self-Healing）**
Agent A 自愈触发 Agent B 重新执行、B 又触发 C……如果没有全局预算，
很容易出现"整个系统在互相修，没人负责停下"。

- 对策：自愈预算必须是**全局共享**的，不是每个 Agent 各有一份
- 对策：跨 Agent 触发的重试算作上游的"一次自愈次数"，避免预算被绕过

#### 协调机制的最小集

Level 4（Fleet）真正要跑起来，至少需要以下协调能力，缺一个都会
在规模上崩盘：

1. **任务分发器**：知道哪个 Agent 在跑什么、资源是否冲突、能否并行
2. **全局 Trace 聚合**：多 Agent 的 Trace 在同一时间轴可查询
3. **统一升级通道**：任何 Agent 要升级 Gate 2 都走同一个入口，
   不允许每个 Agent 各自弹通知
4. **仲裁机制**：当两个 Agent 给出矛盾结论（例如 Implementer 说
   完成、Tester 说未过）时，由预先定义的仲裁方（独立 Agent 或人）
   拍板，而不是让双方"再协商一轮"

没有这些的"多 Agent 协作"其实只是"几个独立 Agent 并排跑"——
看起来吞吐高，实际上事故归因和恢复成本同步上升。

## 7. Harness 的四层模型

到目前为止我们一直在用"Harness"这个词描述整套机制。本章把它
拆成四层，让团队在搭建时知道"我现在在做哪一层、这一层的产物
是什么"。这四层从底向上、相互依赖：

```
┌──────────────────────────────────────────────┐
│ ④ Autonomy Harness   "AI 在哪些边界内可以自治？"
├──────────────────────────────────────────────┤
│ ③ Workflow Harness    "AI 应该按什么顺序做事？"
├──────────────────────────────────────────────┤
│ ② Eval Harness        "AI 做出来的东西怎么算对？"
├──────────────────────────────────────────────┤
│ ① Prompt Harness      "AI 在每一步该怎么思考与表达？"
└──────────────────────────────────────────────┘
```

### 7.1 第 ① 层：Prompt Harness

**关注点**：让 AI 在每一次单独调用中给出**风格一致、格式可解析、
推理可见**的产出。

构成：
- 角色与目标的清晰描述
- 思维链 / 推理可见性的要求
- 输出 schema（让产出可被下游脚本消费）
- few-shot 例子（覆盖典型场景与边界）
- 失败时的 fallback 协议（"如果你不知道怎么做，输出 X"）

Prompt Harness 是基础。没有它，上面三层全部失效——因为你拿到的
每条 AI 输出都是一团需要"再读一遍"的散文，根本谈不上自动化。

### 7.2 第 ② 层：Eval Harness

**关注点**：让"AI 做得对不对"可以被**客观判定**，且判定本身
也可被自动化。

构成：
- 不同层级的验证套件（§4.4 的 L0~L5）
- 黄金集 / 回归集 / 红队集
- 自动评分器（rubric / critic / LLM-as-judge）
- 评估报告的结构化输出格式

Eval Harness 决定了 Phase 2 的"自验证"能不能成立。它的覆盖度
等于 AI 自治的"长度"——能客观判定到 L3，AI 就敢自治到 L3；
只能判定到 L1，那么 L2 以上必须打断人。

### 7.3 第 ③ 层：Workflow Harness

**关注点**：让"多步任务的执行顺序、依赖、回滚"被**显式编排**，
而不是依赖单次 prompt 一气呵成。

构成：
- 阶段定义与转移条件（本文档的三阶段就是 Workflow Harness 的实例）
- 子任务依赖图
- 状态机（每步的"开始 / 完成 / 失败 / 重试 / 回滚"）
- 跨步骤的产物传递与持久化

没有 Workflow Harness 的 AI 表现成"上下文记忆有限的自由叙述者"——
它会重复做、漏做、做反顺序。Workflow Harness 把"它必须按这个顺序、
带这些产物、走这些分支"显式化。

### 7.4 第 ④ 层：Autonomy Harness

**关注点**：让"AI 在没有人在场时可以独立工作"——这是本文档的核心贡献。

构成（与 Ch4 八大支柱直接对应）：
- 计划先行
- 持续验证
- 信心阈值
- 风险分层
- 自愈循环
- 权限作用域
- 环境逐级推进
- 可观测轨迹

外加：
- AutoPilot Report 的结构化标准
- Gate 2 的触发规则
- 自动回滚机制

Autonomy Harness 是"流水线之上的元层"——它决定 Workflow 在多大
范围内可以"自动跑完"。前三层让 AI **能够**做事；第四层让团队
**敢于**让 AI 独立做事。

### 7.5 四层之间的依赖

四层是严格自下而上依赖的：

- 没有 Prompt Harness → Eval 拿不到结构化输出，没法自动评估
- 没有 Eval Harness → Workflow 的"步骤完成"无客观判据，状态机失效
- 没有 Workflow Harness → Autonomy 不知道该自治哪些"段落"，
  无从划定边界

新团队的常见错误是"直接做 Autonomy"——给 AI 一个大目标、
让它自己跑——结果往往是失败或失控。**正确的搭建顺序是从 ① 到 ④
逐层补齐**，每补一层，AI 的可信任范围向上扩张一层。

### 7.6 四层与三阶段的对照

| | Phase 1 Co-Design | Phase 2 Autonomous Build | Phase 3 Validation |
|---|---|---|---|
| ① Prompt | 提问模板、Spec 写作模板 | 决策、自愈、计划的 prompt | Report 模板 |
| ② Eval | Spec 完备性 lint | L0~L5 验证套件 | Gate 3 核对清单 |
| ③ Workflow | ask_user → 初稿 → 修订 → 定稿 | 八大支柱编排 | 验收 → 监控 → 闭环 |
| ④ Autonomy | （N/A，本阶段允许打断） | 八大支柱的护栏 | 自动回滚 |

注意 Phase 1 在 Autonomy 列是 N/A——Co-Design 阶段本就允许甚至
鼓励打断人，自治不是它的目标。Autonomy Harness 主要服务 Phase 2，
辅助 Phase 3（自动回滚、自动监控值守）。

## 8. 反模式（Anti-Patterns）

正面原则讲完了，反面案例有时更有指导意义。下面列出实践中反复出现
的"看似合理但会摧毁 Harness"的反模式，每条配上根因与替代方案。

### 8.1 "AI 当补全工具用"

**症状**：Spec 仍然是模糊的 PRD，AI 主要在写小段代码补全，
人审每一个 commit。

**根因**：没有 Phase 1 的概念，把 Co-Design 跳过了。

**替代**：先把一份功能的 Spec 用 §3.3 的七大部分填齐，
体验一次"AI 自治 → Gate 3 验收"的完整路径。

### 8.2 "Spec 是 PRD 的换名字"

**症状**：Spec 里大段自然语言、没有接口契约、DoD 写"用户体验良好"。

**根因**：没把"机器可读"当成 Spec 的硬要求。

**替代**：建立 Spec 模板的 lint 规则——任何缺机器可解析段落、
缺可机器判定 DoD 的 Spec，CI 直接拒绝定稿。

### 8.3 "Phase 2 期间人忍不住一直看"

**症状**：AI 一开始自治，团队就盯着屏幕、随时介入修一两行。

**根因**：对 Phase 2 不信任，本质是 Eval/Workflow Harness 没建好。

**替代**：找一个低风险任务做"全程不看"的练习，让团队亲身体验
"AI 自己能跑完"。一次成功体验比十次原则讨论更能改变习惯。

### 8.4 "Gate 太多 → 人成瓶颈"

**症状**：每隔一个子任务就要人确认一下，AI 自治退化为"半自动 + 频繁打断"。

**根因**：信心阈值过高、风险分层过严，或团队心理上不放心。

**替代**：诚实评估——Gate 多究竟是 Spec 不够清楚，还是仅仅心理依赖？
前者回 Phase 1 补；后者通过先做小任务建立信任。

### 8.5 "Gate 太少 → 高风险动作裸奔"

**症状**：AI 直接改 schema、直接打生产，没有任何升级。

**根因**：风险分层缺失，permission scope 没设好。

**替代**：Risk Tier T3/T4 必须强制走 Gate 2，权限作用域用工具层
hard-deny 实现，不依赖 AI 自觉。

### 8.6 "AI 声称完成而无验证产物"

**症状**：AutoPilot Report 里写"测试已通过"，但没有报告链接、
没有产物哈希。

**根因**：Continuous Verification 没强制留产物。

**替代**：所有"完成"声明必须附结构化证据（报告 URL、产物 hash、
日志片段），否则视为未完成。

### 8.7 "AI 自由叙述地写 Report"

**症状**：AutoPilot Report 是 AI 自己写的"作文"，描述
"我做了什么、感觉良好"。

**根因**：Report 没有从 Trace 自动汇总。

**替代**：Report 由模板从 Trace 渲染生成，AI 不允许"扩写"，
只允许在固定字段里填数据。

### 8.8 "无限自愈"

**症状**：AI 在某个失败点反复尝试、几十次都没修好，没人知道。

**根因**：自愈预算未设或过松。

**替代**：单点 N=3、总预算 M=10、时间 K=2 是合理起点。任何预算
耗尽立刻升级 Gate 2。

### 8.9 "灰度只看技术指标"

**症状**：CPU/延迟正常 → 全量推。结果业务转化率掉了 20%。

**根因**：监控只盯了技术 SLO，没盯业务 SLI。

**替代**：Spec 阶段就把"成功的业务指标 + 阈值"显式写出，
灰度阶段必须监控这些。

### 8.10 "Gate 3 通过即下线监控"

**症状**：人签字 → 监控关闭 → 第二天线上事故没人知道。

**根因**：把"完成"理解成"签字时"而不是"监控窗口结束时"。

**替代**：Spec 显式定义"监控窗口期"，期内 AI 持续值守、
任一指标越线自动回滚。

### 8.11 "Spec 与代码 drift"

**症状**：三个月后，Spec 库与代码完全不一致，AI 拿到的 Spec
其实是过期文档。

**根因**：缺 §6.2 的双向 drift 检测。

**替代**：CI 强制 PR 声明"是否影响 Spec"、定期反向扫描比对、
不一致进 Phase 1 闭环。

### 8.12 "私底下改 prompt 不通知"

**症状**：某天 AI 行为突变，没人知道为什么——其实是有人改了
全局 prompt 文件没同步给团队。

**根因**：prompt 没版本化、没 review、没 changelog。

**替代**：prompt 与代码同等待遇，进 git、走 PR、有 changelog。

### 8.13 "护栏只装一半"

**症状**：八大支柱只做了 4 个，团队感觉"已经很正规了"。

**根因**：误以为 Harness 是"做一点是一点"的渐进物。

**替代**：对 Level 3+ 的完整自治来说，八大支柱确实是一套完整最低集；
但在采纳路径上，可以逐步建设其中的一部分。关键不是"能不能先做 4 个"，
而是**做了 4 个时就老老实实把自己当成过渡态**，不要把半套护栏误说成
"已经可以 full autonomy"。真正危险的不是渐进建设，而是错判成熟度。

## 9. 成熟度路径

不是所有团队都能一步建成完整 Harness。这一章给出从 Level 0 到
Level 4 的渐进路径：每一级有清晰的"特征 + 应该具备的能力 +
可处理的任务范围"，团队可以诚实地自评定位、有序提级。

在进入分级之前，先讲清一件容易被误解的事：**前文对八大支柱的要求，
描述的是 Level 3+ 的完整自治终态；本章讨论的是你怎样从更低成熟度走到那里。**
两者并不矛盾。矛盾只会出现在另一种情况：团队明明还在 Level 1/2，
却因为已经做了几项护栏，就把自己当成 Level 3。

### 9.1 Level 0 · 补全工具（Assist）

**特征**：AI 用作 IDE 内补全、问答助手；人写代码、人 review。

**典型能力**：
- AI 辅助生成片段
- 问答式协助调试

**任务边界**：所有 PR 由人主导，AI 介入粒度 < 100 行。

**风险**：低；上限：低。

**升级条件**：团队开始觉得"AI 可以独立完成一些小任务"。

### 9.2 Level 1 · 任务执行（Task）

**特征**：能把"明确的小任务"整体丢给 AI（一个 PR、一个 bug 修复），
但 Spec 仍是非结构化的口头描述。

**应具备**：
- 基础 Prompt Harness（输出格式、推理可见）
- 基础 Eval Harness（至少 L0~L1 自动化）
- PR 模板包含基本检查项

**任务边界**：单 PR、单模块、低风险（T0~T1）。

**升级条件**：希望 AI 处理跨模块、能跑完 CI 的任务。

### 9.3 Level 2 · 流程内自动（Workflow）

**特征**：AI 能按显式 workflow 跑多个步骤（生成代码 → 写测试 →
跑 CI → 修缺陷 → 提 PR），但 Spec 仍以 PRD 形式存在，
人在多个节点 review。

**应具备**：
- Workflow Harness 成型（多步编排、状态机、产物传递）
- Eval 扩展到 L2~L3
- 失败有简单重试机制
- Trace 开始结构化

**任务边界**：单功能、跨模块、中风险（T0~T2）。

**升级条件**：开始追求"人在 Phase 2 中尽量不打断"的体验。

#### 从 Level 2 走向 Level 3 的最低可行 Harness（MVP）

很多团队卡在这里：知道完整模型很理想，但不知道第一步该补什么。
一个现实的 MVP 不是八大支柱一次到位，而是先补齐**足以支撑有限自治**
的最小闭环：

- 一份结构化 Spec 模板，至少明确意图、非目标、接口契约、DoD、Open Questions
- 一个真正生效的 Gate 1：Spec 未定稿不得进入 Phase 2
- 一组基础验证：至少 L0~L1，再加一个与任务类型最相关的更高层验证
- 一个粗粒度风险层：哪怕先只有 T0 / T1 / T4，也比完全没有强
- 一个有预算上限的自愈规则：没有预算就宁可先不开自愈
- 一个最小 Trace：先记录关键决策、验证结果、升级与回滚事件

达到这个 MVP，并不意味着你已经拥有完整的 Autonomous Build；
它只意味着你开始具备了**在有限任务上试运行自治**的资格。真正进入
Level 3，仍然要求八大支柱整体成形。

### 9.4 Level 3 · 阶段自治（Autonomous Build）

**特征**：本文档描述的完整模型——三阶段、三 Gate、八大支柱
全部到位。AI 在 Phase 2 默认零打断，遇到 T4 或低信心才升级。

**应具备**：
- Spec 是机器可读契约
- Phase 1 有 ask_user 模板与 Spec 定稿 lint
- Phase 2 八大支柱齐备
- AutoPilot Report 自动从 Trace 汇总
- Gate 3 是"核对事实"而非"通读代码"
- Spec 与代码的 drift 检测在 CI 中

**任务边界**：完整功能、跨模块、含部署与监控（T0~T3 自治、
T4 升级）。

**升级条件**：希望多功能并行、跨任务学习、Harness 自身持续优化。

### 9.5 Level 4 · 组织级 AI 工厂（Fleet）

**特征**：多个 AI 同时跑多个功能，Harness 自身被持续度量、
持续改进；Spec 模板、风险层、阈值、prompt 都是"被进化的资产"。

**应具备**：
- 多 Agent 并行的协调机制（资源、依赖、冲突解决）
- 跨任务的 metric 仪表盘（成功率、Gate 2 率、自愈率、回滚率）
- 失败案例驱动 Spec 模板演化
- prompt / 阈值 / 策略全部版本化、A/B 化
- 有专人持续维护 Harness 本身
- 团队级"AI 行为白皮书"沉淀

**任务边界**：组织级别——多个团队的多个功能并行自治，
人的注意力主要花在 Spec 与 Harness 改进。

### 9.6 提级建议

- **不要跨级跳**：Level 1 直接冲 Level 3 几乎一定失败——
  Eval/Workflow Harness 没有，Autonomy 没有依托
- **每提一级先锚定一个里程碑功能**：用一次完整成功证明这级跑通，
  再把模式向团队复制
- **失败要诚实回退**：Level 3 跑不动就退回 Level 2 加固
  Eval/Workflow，而不是用人力硬撑
- **Level 4 不是必经**：很多团队稳定在 Level 3 已经获得绝大部分价值；
  Level 4 的边际收益与维护成本要算账

### 9.7 自评清单（可直接抄做团队 review）

- [ ] 我们有显式的 Spec 模板，包含七大部分
- [ ] Spec 中接口契约 / DoD 是机器可校验的
- [ ] 我们有 Phase 1 的 ask_user 模板与 Spec 定稿 lint
- [ ] 我们的代码 / PR / 部署 / 监控通过 Feature ID 全链贯通
- [ ] Phase 2 的执行有显式计划与依赖图
- [ ] 我们有 L0~L5 中至少 L0~L3 的自动化验证
- [ ] AI 决策点有信心阈值机制
- [ ] 我们有显式的 Risk Tier 与 permission scope
- [ ] 自愈循环有明确预算上限
- [ ] 部署是逐级环境推进且支持自动回滚
- [ ] 所有 AI 行为有结构化 Trace
- [ ] AutoPilot Report 由模板从 Trace 汇总，不是 AI 自由叙述
- [ ] Gate 3 是核对事实，不再通读代码 / 跑全量回归
- [ ] Spec 与代码有双向 drift 检测
- [ ] Prompt / 阈值 / 策略与代码同等版本化
- [ ] Phase 3 的发现有结构化回流到 Phase 1 模板

打勾比例对应大致级别：
- < 30% → Level 1 或更低
- 30%~60% → Level 2
- 60%~90% → Level 3
- > 90% 且有持续改进机制 → Level 4

## 附录 A · 术语表

| 术语 | 含义 |
|------|------|
| **Harness** | 一套约束 AI 行为、验证 AI 产物、编排 AI 流程、护航 AI 自治的工程体系。本文档把它分为四层：Prompt / Eval / Workflow / Autonomy。 |
| **Spec** | 机器可读、结构化、版本化的需求契约。Phase 1 的产物，Phase 2 的输入，Phase 3 的验收标准。 |
| **Spec-Then-Build with Gates** | 本文档定义的核心模型：Co-Design → Autonomous Build → Validation 三阶段，Gate 1/2/3 三关卡。 |
| **Co-Design (Phase 1)** | 人与 AI 协同收敛 Spec 的阶段，鼓励 ask_user。 |
| **Autonomous Build (Phase 2)** | AI 在 Spec 约束下独立执行的阶段，默认零打断。 |
| **Validation (Phase 3)** | 人核对 AutoPilot Report、系统进入持续监控的阶段。 |
| **Gate 1** | Spec 定稿门禁，必经。 |
| **Gate 2** | Phase 2 内部高风险升级门禁，按需触发，应罕见。 |
| **Gate 3** | 终点验收门禁，必经，但只核对事实不通读代码。 |
| **Feature ID** | 一个功能的全链路追溯锚，贯穿 Spec/PR/commit/测试/部署/监控。 |
| **DoD (Definition of Done)** | 一组可机器判定的完成条件，写在 Spec 里。 |
| **AutoPilot Report** | Phase 2 结束时由 AI 从 Trace 汇总的结构化执行报告。 |
| **Confidence Threshold** | 信心阈值。AI 决策信心低于此值时升级 Gate 2。 |
| **Risk Tiering** | 风险分层。把动作按"出问题代价"分 T0~T4，不同层走不同护栏。 |
| **Self-Healing Loop** | 自愈循环。失败优先尝试自动修复，有预算上限，预算耗尽升级。 |
| **Permission Scope** | 权限作用域。AI 在代码 / 环境 / 外部资源上的最小必要能力集。 |
| **Gradual Env Promotion** | 环境逐级推进。dev → staging → 灰度 → 全量，每级有入门 / 晋级条件。 |
| **Observable Trace** | 可观测轨迹。AI 所有动作、决策、产物的结构化、不可篡改记录。 |
| **Spec Drift** | Spec 与代码失同步的状态。需要双向 detection 持续对冲。 |
| **HITL (Human-in-the-Loop)** | 人在回路。本文档主张少量关键 HITL Gate，而非全程参与。 |
| **Context Engineering** | Context 工程。让 AI 在每个决策点恰好看到该看的信息、避免噪声淹没。 |
| **SLO / SLI** | Service Level Objective / Indicator。SLI 是客观测量的指标（错误率、p95 延迟等），SLO 是这些指标的目标阈值。本文档在灰度监控与 DoD 中大量使用。 |
| **Rubric / LLM-as-judge** | 两种自动评分手段。Rubric 是预先定义的打分细则（可被脚本判定）；LLM-as-judge 是用另一个 LLM 依照 rubric 打分，适合难以写死规则的主观产物（如文案）。 |
| **Red Team 集** | 红队集。一组刻意构造来"击穿"系统的用例——包含 prompt 注入、越权、边界输入等，用于验证 Harness 是否在对抗环境下仍能守住边界。 |
| **Arbiter Agent** | 仲裁 Agent。多 Agent 协作时负责在冲突决策间拍板的独立 Agent；职责固定、上下文隔离，避免成为"又一个利益相关方"。 |

## 附录 B · 业界资料引用与延伸阅读

本文档的方法论建立在以下业界共识与资料之上。它们不是"原文照抄"，
而是"经过提炼、对齐、整合后的方法论"——读者可以按需检索原文获得
更详细的背景。

### B.1 Spec-Driven Development

- 主题：将 Spec 作为 single source of truth、机器可读、纳入版本管理
- 代表实践：GitHub Spec Kit、AWS 类似实践、AI 编辑器对 spec 文件的一等支持
- 与本文档对应：Ch3 全章、Ch6.1~6.2

### B.2 HITL Gate Pipeline

- 主题：在 AI 流水线中只设少量关键 HITL Gate，而非全程介入
- 触发条件常见来源：歧义、安全合规、高风险、低信心、异常
- 检索关键词：`agentic SDLC`、`HITL gate`、`confidence-triggered escalation`
- 与本文档对应：Ch2.3、Ch4.5、Ch4.6

### B.3 Cognition / Devin 八条原则（提炼版）

1. 透明可追溯的推理（transparent reasoning）
2. 持续验证（continuous verification）
3. 信心阈值触发 HITL（HITL at confidence thresholds）
4. 目标跟踪与对齐（goal tracking）
5. 权限作用域 + 渐进环境推进（permission scopes + gradual promotion）
6. 主动反馈（active feedback）
7. 部署后监控 + 自动回滚（post-deploy monitoring + auto-rollback）
8. 可复现性（reproducibility）

- 与本文档对应：Ch4 八大支柱（一一对照、本文档表述更工程化）

### B.4 Anthropic Claude Code 模式

- 主题：plan-first、TDD、增量编辑、`CLAUDE.md` / `AGENTS.md` 作为
  仓库级 context 锚点
- 与本文档对应：Ch4.3（Plan-First）、Ch6.3（Context 工程）

### B.5 Agentic 系统的工程边界

- 主题：明确能力边界、最小权限、可审计、失败时的 fallback 与
  graceful degradation、self-monitoring、retry + backoff、idempotency
- 检索关键词：`LLM agent reliability`、`agent permission scopes`、
  `self-monitoring agent`、`idempotent tool use`
- 与本文档对应：Ch4.6~4.10 的多个支柱

### B.6 AI Native SDLC（2025 业界共识）

- 主题：AI Native 不是"传统 SDLC + AI 工具"，而是把 AI 当成
  default executor、把人放在 Spec/Gate 上的根本性流程重构
- 与本文档对应：Ch2.4 与传统 SDLC 的对比、Ch9 成熟度路径

### B.7 推荐进一步阅读方向

- "Specification-driven AI development" 类博客与开源工具
- "Agentic coding pipelines" 与 "CI/CD for AI agents" 主题
- "LLM observability"、"AI eval harness"、"red-teaming"
- "Risk tiering for autonomous systems" 在自动驾驶等领域的迁移借鉴
- "Self-healing software" 与 "auto-rollback" 在 SRE 领域的成熟实践

> 业界正处于快速演化期，本文档的方法论是 2025 年时点的最佳归纳。
> 强烈建议团队定期回看上述方向，把新出现的模式回灌到自己的
> Spec 模板、风险分层、阈值与 prompt 中——**这正是 §5.4 所说的
> 闭环在 Harness 自身上的体现**。
